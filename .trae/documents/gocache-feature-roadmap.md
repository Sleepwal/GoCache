# GoCache 功能开发路线图 (3-6 个月)

> 制定日期: 2026-04-17 | 当前版本: v1.0.0 | 目标版本: v2.0.0

***

## 一、项目现状评估

### 已完成功能清单

| 模块   | 功能                          | 状态     |
| ---- | --------------------------- | ------ |
| 核心缓存 | CRUD、TTL、惰性删除+定期清理          | ✅ 稳定   |
| 淘汰策略 | LRU、LFU (含时间衰减)             | ✅ 稳定   |
| 数据结构 | String、List、Hash、Set        | ✅ 稳定   |
| 持久化  | Snapshot (JSON/Gob)、AOF     | ✅ 基础完成 |
| 服务层  | HTTP REST API               | ✅ 基础完成 |
| 辅助功能 | 命名空间、Pub/Sub、统计指标、泛型缓存、内存限制 | ✅ 稳定   |
| 日志   | 异步结构化日志系统                   | ✅ 稳定   |

### 当前架构短板

1. **单机限制** — 无集群/分布式能力，无法水平扩展
2. **协议单一** — 仅 HTTP REST，缺少高性能二进制协议
3. **运维缺失** — 无 CLI 工具、无 Web 管理界面、无监控集成
4. **数据结构不足** — 缺少 Sorted Set、Bitmap、HyperLogLog 等高级结构
5. **持久化薄弱** — AOF 无自动重写触发、无 RDB 式快照策略、无混合持久化
6. **安全空白** — 无认证、无权限控制、无 TLS
7. **事务缺失** — 无 MULTI/EXEC、无 Lua 脚本支持

***

## 二、功能模块规划与优先级

### 优先级定义

* **P0 (关键)**: 核心竞争力，必须完成

* **P1 (重要)**: 显著提升可用性，应优先完成

* **P2 (增强)**: 提升用户体验和生态，按资源情况推进

* **P3 (远期)**: 长期战略布局，可延后

***

### 模块 1: Sorted Set 数据结构 (P0)

**理由**: Sorted Set 是排行榜、延迟队列等场景的核心数据结构，是 Redis 最重要的数据类型之一，缺失将严重限制应用场景。

| 子功能                       | 优先级 | 开发周期 | 依赖     |
| ------------------------- | --- | ---- | ------ |
| ZADD/ZREM/ZCARD           | P0  | 3 天  | 无      |
| ZRANGE/ZREVRANGE (按分数范围)  | P0  | 3 天  | 跳表实现   |
| ZRANK/ZREVRANK (按排名)      | P0  | 2 天  | 跳表实现   |
| ZINCRBY/ZSCORE            | P0  | 1 天  | 无      |
| ZUNIONSTORE/ZINTERSTORE   | P1  | 3 天  | ZADD   |
| ZPOPMIN/ZPOPMAX           | P1  | 1 天  | 跳表实现   |
| ZRANGEBYLEX/ZRANGEBYSCORE | P1  | 2 天  | ZRANGE |
| HTTP API 适配               | P0  | 2 天  | 上述全部   |

**技术方案**: 基于跳表 (Skip List) + 哈希表实现，O(logN) 的插入/删除/排名查询。

**验收标准**:

* 所有操作单元测试覆盖率 ≥ 90%

* 基准测试: ZADD ≥ 50万 ops/s, ZRANGE ≥ 20万 ops/s

* HTTP API 完整可用

**风险**: 跳表实现复杂度中等，需仔细处理并发安全。

***

### 模块 2: Redis 协议兼容层 (P0)

**理由**: Redis 协议 (RESP) 是缓存/内存数据库的事实标准，兼容 RESP 可直接复用 redis-cli、各种语言的 Redis 客户端库，极大降低用户迁移成本。

| 子功能                     | 优先级 | 开发周期 | 依赖       |
| ----------------------- | --- | ---- | -------- |
| RESP 协议解析器              | P0  | 4 天  | 无        |
| RESP 响应编码器              | P0  | 2 天  | 无        |
| TCP 服务器框架               | P0  | 3 天  | 无        |
| 命令路由框架                  | P0  | 2 天  | RESP 解析器 |
| 基础命令适配 (SET/GET/DEL 等)  | P0  | 3 天  | 命令路由     |
| String 命令适配             | P0  | 2 天  | 基础命令     |
| List 命令适配               | P0  | 2 天  | 基础命令     |
| Hash 命令适配               | P0  | 2 天  | 基础命令     |
| Set 命令适配                | P0  | 2 天  | 基础命令     |
| Sorted Set 命令适配         | P1  | 2 天  | 模块 1     |
| Pub/Sub 命令适配            | P1  | 2 天  | 基础命令     |
| 连接管理 (AUTH/SELECT/QUIT) | P1  | 2 天  | TCP 服务器  |

**技术方案**: 实现 RESP2 协议解析/编码，TCP 长连接 + 命令分发模式。

**验收标准**:

* 通过 redis-cli 完整交互测试

* 至少 3 种语言客户端 (go-redis, jedis, redis-py) 可正常连接操作

* 并发连接 ≥ 1000，单连接 QPS ≥ 10万

**风险**: 协议细节多，需大量兼容性测试；Pipeline 模式实现需注意性能。

### 模块 3: CLI 命令行工具 (P1)

**理由**: 类似 redis-cli 的交互式命令行工具是开发者日常运维的必备工具，直接影响开发体验。

| 子功能               | 优先级 | 开发周期 | 依赖   |
| ----------------- | --- | ---- | ---- |
| 交互式 REPL          | P1  | 3 天  | 无    |
| 命令补全/提示           | P1  | 2 天  | REPL |
| 输出格式化 (表格/JSON)   | P1  | 2 天  | 无    |
| 单次命令执行模式          | P1  | 1 天  | REPL |
| 管道模式 (stdin 批量执行) | P2  | 2 天  | 单次命令 |
| 集群模式支持            | P3  | 3 天  | 模块 7 |

**技术方案**: 使用 Go 标准库 `bufio.Scanner` 实现 REPL，`github.com/olekukonez/tablewriter` 实现表格输出。

**验收标准**:

* 支持所有已实现命令的交互式执行

* 命令补全覆盖 ≥ 80% 命令

* 支持 `gocache-cli SET key value` 单次执行模式

**风险**: 低，技术成熟。

### 模块 4: 持久化增强 (P1)

**理由**: 当前 AOF 仅支持手动 Rewrite，无自动触发机制；缺少 RDB 式定时快照；混合持久化可兼顾恢复速度和数据安全。

| 子功能                               | 优先级 | 开发周期 | 依赖          |
| --------------------------------- | --- | ---- | ----------- |
| AOF 自动重写触发 (按文件大小/时间)             | P1  | 3 天  | 现有 AOF      |
| AOF fsync 策略 (always/everysec/no) | P1  | 2 天  | 现有 AOF      |
| RDB 定时快照 (fork 子进程)               | P1  | 5 天  | 现有 Snapshot |
| 混合持久化 (RDB+AOF)                   | P2  | 5 天  | RDB + AOF   |
| 增量快照 (仅保存变更)                      | P2  | 4 天  | RDB         |
| 持久化配置热更新                          | P2  | 2 天  | 上述全部        |

**技术方案**:

* AOF 自动重写: 监控文件大小增长比例，超过阈值自动触发 `Rewrite`

* RDB 快照: 使用 `runtime.LockOSThread` + fork 或序列化到临时文件后原子替换

* 混合持久化: RDB 格式写入 + 增量 AOF 追加

**验收标准**:

* AOF 自动重写: 文件增长超过 200% 自动触发，重写期间不阻塞读写

* RDB 快照: 100万 key 快照时间 ≤ 5 秒

* 异常重启后数据零丢失 (fsync=everysec 模式)

**风险**: RDB fork 在 Windows 上不可用，需考虑跨平台兼容方案 (序列化到临时文件)。

***

### 模块 5: 事务与脚本支持 (P1)

**理由**: 事务是保证数据一致性的基础能力，Lua 脚本支持可提供原子性复杂操作，是 Redis 的核心差异化能力。

| 子功能                      | 优先级 | 开发周期 | 依赖         |
| ------------------------ | --- | ---- | ---------- |
| MULTI/EXEC 事务            | P1  | 4 天  | 无          |
| WATCH 乐观锁                | P1  | 3 天  | MULTI/EXEC |
| DISCARD 取消事务             | P1  | 1 天  | MULTI/EXEC |
| Lua 脚本引擎集成               | P2  | 5 天  | gopher-lua |
| EVAL/EVALSHA 命令          | P2  | 3 天  | Lua 引擎     |
| SCRIPT LOAD/EXISTS/FLUSH | P2  | 2 天  | EVAL       |

**技术方案**:

* 事务: 命令队列 + 顺序执行，WATCH 通过版本号检测冲突

* Lua: 集成 `github.com/yuin/gopher-lua`，提供缓存操作 API

**验收标准**:

* 事务内操作原子执行，WATCH 检测到冲突时自动回滚

* Lua 脚本可访问所有数据结构操作

* 脚本执行超时保护 (默认 5 秒)

**风险**: Lua 引擎引入外部依赖，需评估性能和安全性；脚本沙箱隔离需谨慎处理。

***

### 模块 6: 安全与认证 (P1)

**理由**: 生产环境部署必须具备基本的认证和访问控制能力，否则无法安全暴露服务。

| 子功能                     | 优先级 | 开发周期 | 依赖         |
| ----------------------- | --- | ---- | ---------- |
| 密码认证 (AUTH)             | P1  | 2 天  | 无          |
| ACL 权限控制 (用户+命令+Key 级别) | P2  | 5 天  | AUTH       |
| TLS 加密传输                | P1  | 3 天  | 无          |
| 配置文件支持                  | P1  | 3 天  | AUTH + TLS |
| 慢查询日志                   | P2  | 2 天  | 无          |
| 审计日志                    | P3  | 3 天  | ACL        |

**技术方案**:

* AUTH: 连接建立后需先认证，支持明文和 SHA256 密码

* ACL: 基于用户+命令模式+Key 模式的三维权限矩阵

* TLS: 使用 Go 标准库 `crypto/tls`

* 配置文件: TOML 格式，支持热重载

**验收标准**:

* 未认证连接无法执行任何操作

* ACL 可限制特定用户只能访问特定 Key 前缀

* TLS 握手性能损耗 ≤ 5%

**风险**: ACL 实现复杂度较高，需仔细设计权限模型。

***

### 模块 7: 集群与分布式 (P2)

**理由**: 分布式能力是从单机缓存走向生产级内存数据库的关键跨越，但实现复杂度最高，需谨慎推进。

| 子功能                | 优先级 | 开发周期 | 依赖   |
| ------------------ | --- | ---- | ---- |
| 一致性哈希分片            | P2  | 5 天  | 无    |
| 集群拓扑管理 (Gossip 协议) | P2  | 7 天  | 分片   |
| 数据迁移 (在线 Reshard)  | P2  | 7 天  | 拓扑管理 |
| 主从复制               | P2  | 5 天  | 无    |
| 哨兵模式 (自动故障转移)      | P3  | 7 天  | 主从复制 |
| 分布式事务 (跨分片)        | P3  | 10 天 | 分片   |

**技术方案**:

* 分片: Hash Slot (16384 槽位) 模式，类似 Redis Cluster

* 复制: 异步复制 + PSYNC 增量同步

* Gossip: 节点间定期交换集群状态信息

**验收标准**:

* 3 节点集群可正常分片读写

* 单节点故障后集群可在 10 秒内完成故障转移

* 在线 Reshard 期间服务不中断

**风险**: 分布式一致性是最大挑战，Gossip 协议实现复杂，需大量混沌测试。

***

### 模块 8: Web 管理界面 (P2)

**理由**: 可视化管理降低运维门槛，提升产品竞争力。

| 子功能                       | 优先级 | 开发周期 | 依赖        |
| ------------------------- | --- | ---- | --------- |
| Dashboard 概览 (内存/命中率/连接数) | P2  | 3 天  | 无         |
| Key 浏览器 (搜索/查看/编辑/删除)     | P2  | 4 天  | 无         |
| 实时监控图表                    | P2  | 3 天  | Dashboard |
| 配置管理界面                    | P2  | 2 天  | 配置文件支持    |
| 慢查询分析界面                   | P3  | 2 天  | 慢查询日志     |
| 集群管理界面                    | P3  | 5 天  | 模块 7      |

**技术方案**: 嵌入式前端 (embed.FS)，后端 API 复用现有 HTTP Server，前端使用 React/Vue 轻量 SPA。

**验收标准**:

* 单二进制部署，无需额外前端服务

* 实时数据刷新延迟 ≤ 1 秒

* 支持 Key 的 CRUD 操作

**风险**: 前端开发资源需求较大，可考虑使用成熟的开源 Redis 管理界面改造。

## 三、时间线与阶段规划

### 阶段一: 核心补强 (第 1-2 个月)

**目标**: 补齐核心数据结构和高性能协议，达到可用级别

```
第 1-2 周: Sorted Set 数据结构 (模块 1)
第 3-4 周: RESP 协议解析器 + TCP 服务器 (模块 2 前半)
第 5-6 周: RESP 命令适配 - 基础/String/List/Hash/Set (模块 2 后半)
第 7-8 周: CLI 工具 + 持久化增强 (模块 3 + 模块 4)
```

**里程碑**: v1.1.0 — 支持 Sorted Set + Redis 协议兼容 + CLI 工具

**验收标准**:

* redis-cli 可完整操作所有数据结构

* gocache-cli 可交互式使用

* AOF 自动重写可用

***

### 阶段二: 生产就绪 (第 3-4 个月)

**目标**: 增强安全、事务、监控能力，达到生产可用级别

```
第 9-10 周: 事务支持 MULTI/EXEC/WATCH (模块 5 前半)
第 11-12 周: 安全认证 AUTH + TLS + 配置文件 (模块 6)
第 13-14 周: 持久化增强 RDB + 混合持久化 (模块 4 后半)
第 15-16 周: Prometheus 监控集成 (模块 10)
```

**里程碑**: v1.5.0 — 事务 + 安全认证 + 增强持久化 + 监控

**验收标准**:

* 事务原子性通过并发测试

* TLS + AUTH 可正常工作

* Prometheus 可抓取完整指标

* 异常重启数据零丢失

***

### 阶段三: 生态扩展 (第 5-6 个月)

**目标**: 分布式能力、高级数据结构、管理界面

```
第 17-19 周: 集群分片 + 主从复制 (模块 7 前半)
第 20-21 周: 更多数据结构 Bitmap/HLL/Geo (模块 9)
第 22-23 周: Web 管理界面 (模块 8)

```

**里程碑**: v2.0.0 — 分布式 + 完整数据结构 + Web 管理

**验收标准**:

* 3 节点集群稳定运行 72 小时无异常

* 所有数据结构通过完整测试套件

* Web 界面可完成日常运维操作

## 六、风险评估

### 高风险

| 风险                      | 影响       | 缓解措施                                |
| ----------------------- | -------- | ----------------------------------- |
| 集群分布式一致性难以保证            | 数据丢失/不一致 | 先实现主从复制，再逐步演进集群；充分混沌测试              |
| RESP 协议兼容性不完整           | 客户端库无法使用 | 优先兼容 redis-cli 和 go-redis；建立兼容性测试矩阵 |
| Windows 平台 RDB fork 不可用 | 持久化功能受限  | 实现序列化到临时文件的替代方案                     |

### 中风险

| 风险           | 影响         | 缓解措施                      |
| ------------ | ---------- | ------------------------- |
| Lua 脚本安全沙箱逃逸 | 安全漏洞       | 限制脚本可访问的 API；设置执行超时       |
| 性能回归         | 新功能影响核心性能  | 每个版本运行基准测试对比；建立性能基线       |
| API 稳定性      | 接口频繁变更影响用户 | 遵循语义化版本；内部 API 与公开 API 分离 |

### 低风险

## 八、用户反馈与市场考量

### 目标用户画像

1. **Go 开发者** — 需要嵌入式缓存库，偏好纯 Go 实现
2. **中小团队** — 需要轻量级 Redis 替代品，降低运维成本
3. **边缘计算/IoT** — 需要低资源占用的内存数据库
4. **学习/教学** — 理解内存数据库内部实现

### 市场差异化策略

| 维度 | Redis    | GoCache 定位             |
| -- | -------- | ---------------------- |
| 部署 | 独立服务     | 嵌入式 + 独立服务双模式          |
| 依赖 | C 语言，需编译 | 纯 Go，单二进制              |
| 协议 | RESP     | RESP + HTTP REST (双协议) |
| 扩展 | C 模块     | Go 插件 (更易扩展)           |
| 资源 | 较高       | 轻量级，适合边缘场景             |

### 用户反馈优先处理

1. **RESP 协议兼容** — 最高频需求，直接决定能否替代 Redis
2. **Sorted Set** — 排行榜场景刚需
3. **CLI 工具** — 日常开发效率关键
4. **安全认证** — 生产部署前提条件
5. **监控集成** — 云原生部署必备

## 九、技术债务与重构计划

在推进新功能的同时，需同步处理以下技术债务:

| 项目                                                          | 优先级 | 计划时间 |
| ----------------------------------------------------------- | --- | ---- |
| `evictOne()` 当前为随机淘汰，应改为 LRU 淘汰                             | P1  | 阶段一  |
| PubSub 订阅 ID 使用 `string(rune(counter))`，大数量时会冲突             | P1  | 阶段一  |
| HTTP Server 路由使用手动字符串匹配，应改用路由框架                             | P2  | 阶段二  |
| 缺少配置文件支持，所有参数硬编码                                            | P1  | 阶段二  |
| 日志系统尚未集成到缓存核心模块                                             | P2  | 阶段一  |
| 基准测试需建立持续追踪机制                                               | P2  | 阶段一  |
| `ListCache`/`HashCache`/`SetCache` 内部各自创建 MemoryCache，应支持共享 | P1  | 阶段一  |
